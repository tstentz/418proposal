Title

PQ Unlocked: Lock-Free Priority Queue for Use in A*

Summary

We are going to implement a lock-free priority queue. In addition, we will use this lock-free priority queue to implement a parallel A* algorithm for use within popular games that require finding the shortest path, such as Pacman.

Background

We are interested in lock-free data structures. Many parallel priority queues have been implemented, but they require locks which can slow down insertion and deletion operations. Finding the shortest path quickly is essential for many real time systems, such as autonomous vehicles that need to quickly calculate the correct path from point A to point B. We don't plan on building a driverless car, but we would like to apply the same logic to characters within games. The faster characters can make descisions, the more exciting and challenging the gameplay. We would like to see how different implementations of Priority Queues affect performance of the A* algorithm. To do this, we will build a sequential Priority Queue, a priority queue with coarse-grained locks, and a lock free priority queue.

We plan on implementing the priority queue as a min heap. We also plan on using a skip list so that our PQ can be lock-free. We got this idea and diagram from the paper cited in the Resources Section.

Challenge

This project will pose many challenges along the way. First, we will need to implement several different versions of the priority queue including a coarse-grained locking min heap, a fine-grained queue, and finally a lock-free implementation. By implementing each of these separately, we will better be able to test different workloads in the context of A star to determine if our implementation is correct and provides enough speedup. In A star, each thread will be accessing the priority queue each time it has finished looking at its next node in the frontier. Another challenge is to make sure that our underlying skip list used for our lock free priority queue does not take up too much memory. Because the skip list takes a lot of memory, there may be a high processor-to-processor communication to work ratio in order to retrieve new values in the skip list and remain synchronized. Similarly, each processor will need to constantly retrieve information from memory which may cancel out the benefits of the lock-free data structure.
Finally, for demoing and testing purposes, we will need build an interactive game or solver in Python that will call C++ wrappers that will use our lock free queue and parallel A star interface. Some overall constraints for our project that make mapping the workload challenging is if there are similar workloads in each step of A star, as there will then be high contention for the priority queue. This would result in many threads restarting their dequeue operation over and over.

Resources

Our implementation of the lock-free priority queue will be written from scratch, though we will use tools such as OpenMP to parallelize A*. Additionally, we will use the Python/C library to make calls to our implementation from Python. The final implementation will run on the Intel machines or the Latedays machines. We will be using the following technical report as reference material: Fast and Lock-Free Concurrent Priority Queues for Multi-Thread Systems by Hakan Sundell and Philippas Tsigas

Goals and Deliverables

Plan To Achieve

Our project relies heavily on having a priority queue in which one can simulataneously delete and insert nodes. First, we plan to implement a completely sequential priority queue. Then, we plan to create a priority queue with different granularities of locking for comparison. Finally, we will implement our lock-free priority queue in hopes that it will be able to insert and delete a series of nodes from a graph much faster than the locking and sequential versions of the queue. Our hope is that we can achieve much higher speed-up that locking priority queues even with large supporting data structures like SkipLists. We also hope to better understand why lock-free data structures can be so much faster.

Then, we plan to use our priority queues in a parallel implementation of the A* algorithm.

On top of all this, we will have a test harness to check the correctness and performance of our code from the beginning. We plan on timing average insertion and deletion operation in addition to the overall time it takes to insert and delete a series of elements.

With this plan, we will show speedup graphs to compare the differences in priority queue implementations.

Hope To Achieve

On top of the other plans, we would like to use our A* and lock-free priority queue written in C for a Python version of some game, like Pacman.

As a fun demo, we would like to have the ghosts chase Pacman using the A* algorithm to find him. We would like to run our Pacman three separate times where the ghosts rely on sequential, locking, and lock-free priority queue. We would like to see if the user scores lower on average when the ghosts are able to find the shortest paths faster due to better A* implementations.

We hope to learn whether locking or lock free implementations of a priority queues are better for small and large graphs and if these implementations affect gameplay that relies on A*.

Another moon-shot idea is to extend our application to apply to the D* algorithm, where the graph can change in the middle of running and changes are propagated across certain paths. More about D* can be found here
